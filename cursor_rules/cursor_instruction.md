## Instructions for Cursor Agent (Path A — Rhino)

You are a senior full-stack Rhino developer. When the user asks you to generate, update, or debug features in this project, follow these rules:

### 1. Always Read Documentation First
Before implementing anything:
1. Read `cursor_rules/main.md` for architecture overview and navigation.
2. Read `docs/prd.json` (or `docs/prd.pdf`) for the user's requirements.
3. Reference the specific `cursor_rules/server/` or `cursor_rules/web/` doc relevant to the task.
4. If stuck or need deeper Rhino knowledge, consult `docs/rhino_docs/` for official guides, tutorials, and blog posts.

### 2. PRD Parsing Rules
When working with the PRD, extract these sections correctly:

**Data Model** — found at `current_prd.planning.data_model.planned`:
- Each entry has: `model`, `table`, `columns`, `associations`, `validations`, `rhino_owner`, `provided_by`, `model_content`, `migration_content`, `rails_model_generation_command_line`
- Models with `provided_by` set (e.g., `rhino_project_core`, `rhino_project_organizations`) are framework-provided — do not regenerate them
- Models without `provided_by` are custom — use the `model_content` and `migration_content` as templates

**User Stories** — found at `current_prd.user_stories`:
- Each story has: `title`, `phase`, `key_feature`, `user_story` (with `as_a`, `i_want`, `so_that`), and `acceptance_criteria`
- Use the phase to prioritize (MVP first, then later phases)
- Map acceptance criteria to test cases

**User Journeys** — found at `current_prd.user_journeys`:
- Each journey has `steps` (with `actor` and `description`), `context`, and `trigger`
- Use journeys to design page flows and navigation

**User Personas** — found at `current_prd.user_personas`:
- Each persona has: `name`, `age_role`, `goals`, `pain_points`, `preferred_device`
- Use personas to inform role-based access and UI decisions

**Phasing** — found at `current_prd.phasing`:
- Each phase has: `phase_name`, `features`, `rationale`
- Build MVP phase features first before moving to later phases

**Sitemap and Navigation** — found at `current_prd.sitemap` and `current_prd.navigation_design`:
- Use the sitemap to define routes and page hierarchy
- Use navigation_design for shell layout, sidebar links, and header components

### 3. Code Generation Rules — Always Use Rhino Patterns

**For Backend (Models):**
- Always generate models with `bin/rails g model ModelName ...` first
- Structure models with section comments: `# associations`, `# rhino`, `# callbacks`, `# attributes`, `# enums`, `# validations`, `# scopes`, `# instance methods`
- Set `rhino_owner` and `rhino_references` per `cursor_rules/server/models.md`
- Implement `display_name` if the model has no `name` column
- Follow patterns from `cursor_rules/server/models.md`

**For Backend (Policies):**
- Name policies as `<role>_<resource>_policy.rb`
- Inherit from `::Rhino::ViewerPolicy` or `::Rhino::AdminPolicy`
- Always implement a `Scope` class
- Follow patterns from `cursor_rules/server/policies.md`

**For Backend (Controllers):**
- Keep controllers thin — Rhino handles CRUD automatically
- Only create custom controllers for non-standard actions
- Follow patterns from `cursor_rules/server/controllers.md`

**For Backend (Migrations):**
- Use Rails migration generators
- Follow the migration patterns from the PRD's `migration_content` field

**For Frontend (API Integration):**
- Use Rhino hooks: `useModelIndex`, `useModelShow`, `useModelCreate`, `useModelUpdate`, `useModelDestroy`
- For custom endpoints, use the patterns in `cursor_rules/web/rhino-api.md`
- Do NOT create custom fetch calls when Rhino hooks exist

**For Frontend (Routing):**
- Use TanStack Router
- Routes live in `app/frontend/routes/`
- Authenticated routes go under `app/frontend/routes/_authenticated/`

### 4. Development Workflow
When implementing a feature end-to-end, follow this order:
1. **Model** — Create/update the Rails model with Rhino properties
2. **Migration** — Generate and run the database migration
3. **Policy** — Set up authorization policies
4. **Controller** — Add custom controller actions (if needed; Rhino CRUD is automatic)
5. **Frontend** — Build UI components
6. **Tests** — Write backend (Minitest) and frontend (Vitest/Cypress) tests

### 5. Error Handling Principles
- **Raise errors instead of hiding bugs** — use `find!` not `find_by` when the record must exist
- **No silent failures** — avoid safe navigation (`&.`) when the value should exist
- **Validate inputs early** at boundaries (controllers, service objects)
